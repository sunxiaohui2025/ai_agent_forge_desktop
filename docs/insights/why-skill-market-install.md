# 为什么做「从市场安装技能」

## 解决的用户问题
给专家挂载技能的前提是技能库里得先有技能。此前要新增一个技能，用户得自己在别处找到、下载好 zip 包再上传，或者手写 path/callable/YAML。对「我只是想给这个专家加个 PPT 生成能力」这种诉求来说，链路太长、门槛太高。市场安装把「发现 → 获取 → 入库」收敛成应用内的一次点击，技能库从「自己攒」变成「随用随取」。

## 为什么直连 API 而不是用官方 CLI
腾讯提供了 `skillhub` 命令行工具，看起来是最省事的集成方式。但它会全局装到 `~/.skillhub`、靠 `--dir` 往 OpenClaw 风格的 `skills/` 目录里塞文件、还带自升级逻辑——这些假设都和本产品「打包成桌面应用、自带 FastAPI sidecar、技能落在自己的 SKILLS_DIR 并登记进数据库」的形态冲突。如果包进 CLI，就得在用户机器上额外装一套东西、处理跨平台 PATH、子进程、版本漂移。

我们发现 CLI 底下就是一套干净的公开 HTTP API，于是直接调它。好处是：装到哪、登记什么、怎么扫描，全都由我们自己掌控，dev 和打包构建行为完全一致，用户机器上不留额外痕迹。**与其适配一个面向别的工具链设计的 CLI，不如直接消费它依赖的 API。**

## 为什么复用上传链路而不是另起一套
市场下载回来的包，根目录就有 `SKILL.md`，和用户手动上传的 zip 结构一模一样。既然终点（落盘到 SKILLS_DIR + 入库 atomic Skill + 生成使用说明）完全相同，就没有理由写第二套。我们把上传流程的核心抽成一个共享函数，市场安装只是把「用户选的本地文件」换成「从远端下载的字节流」。这样安全扫描、路径穿越防护、摘要生成这些既有保障，市场安装自动全都享有。

## 安全上的取舍
技能本质是会被智能体执行的代码，来源是公网市场，风险不能只靠"对方说安全"。我们做了双重把关：一是 SkillHub 自带的腾讯安全报告（keen/sanbu）链接照常展示给用户参考；二是**不管来源如何，本地一律重跑自己的静态扫描**，命中风险默认拦截，必须管理员显式勾选「强制安装」才放行。外部背书是参考，本地闸门才是底线。

## 已知边界
市场是「锦上添花」的获取渠道，不是强依赖。内网隔离部署可一键关闭（`SKILLHUB_ENABLED=false`），其余手动上传 / 新建能力不受影响。「检查更新」目前只埋了版本号、没做升级 UI——先把「装得进来」做扎实，升级留给真正有诉求时再做。
