# 为什么给专家加「默认工作目录」

## 解决的用户问题
专家与本地目录的绑定原本只能在「每次对话」层面完成：用户得在首页对话框手动挑一个项目目录，专家才拿得到读写文件、跑命令的能力。但很多专家本质上**长期服务于同一个目录**——某个代码仓库、某份资料库、某个数据落盘目录。让用户每开一个对话就重选一次，既繁琐又容易忘，忘了就退化成"只会聊天、动不了文件"，体验落差很大。

## 为什么这样设计
我们把工作目录做成**三级回退**而不是单一来源：

1. 用户当下在对话框选的目录（临场意图最强，优先级最高）
2. 专家自带的默认目录（稳定、长期、省心）
3. 都没有就传空（诚实退回普通对话，而不是假装能动文件）

这样既保留了用户随时切换目录的灵活性，又让"这个专家就该在这个目录干活"的常见诉求一次配置、长期生效。关键决策是**用户的临场选择永远压过专家默认值**——默认值是兜底，不是枷锁。

## 为什么默认目录不存在时退回普通对话
配置的目录可能被删除、改名，或在另一台机器上根本不存在。与其报错打断用户，不如静默退回普通对话模式（底层用 `isdir` 守卫）。专家少了文件能力，但对话依然可用——可用性优先于严格性。

## 已知局限与未来方向
- 目前 `work_dir` 是单个目录，未来可考虑多目录 / 目录模板（按用户名展开）
- Web 端只能手动粘贴路径，桌面端才有 OS 目录选择器
- 没有对路径做存在性 / 权限的预校验，依赖运行时守卫；后续可在保存时给出"该目录当前不存在"的软提示
