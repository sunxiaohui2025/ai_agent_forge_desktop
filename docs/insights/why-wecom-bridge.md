# 为什么做企业微信桥接 — 产品思考

> 技术实现见 [docs/handover/wecom-bridge.md](../handover/wecom-bridge.md)

## 用户要解决的问题
用户希望"用企业微信的 App 连接我的智能体、可以对话"。本质诉求：**离开电脑、
不开这个桌面应用，也能随手用手机里的企业微信跟自己配好的专家聊一句**——
通勤路上问一句、临时让专家跑个活儿。这是把"桌面里的专家"延伸到"口袋里的 IM"。

## 为什么是企业微信，而不是只做飞书
飞书已经能用，但很多公司/团队的日常 IM 是企业微信。对这部分用户，
"能不能用企业微信"直接决定了远程桥接这个能力对他们**是否存在**。
渠道是乘法不是加法：少一个主流 IM，就少掉一整批用户的入口。

## 关键产品取舍
1. **沿用"渠道目录数据驱动"而不是硬编码 UI。**
   前端 Bridge.vue 完全按后端 `CHANNELS` 渲染，新增渠道只动后端就自动出 Tab。
   代价是前端无法对单渠道做特别花哨的引导，但换来"加渠道近乎零前端成本"，
   对一个要持续接入 QQ/钉钉/公众号的功能，这个一致性远比个性化重要。

2. **接受"企业微信需要公网回调"这个硬约束，并把它讲清楚。**
   飞书能长连接是它的特例，企业微信只有加密 webhook。与其假装能像飞书一样
   零配置，不如老实在 UI 里告诉用户"回调需公网可达，桌面请用 cpolar/frp 穿透"。
   含糊其辞只会让用户卡在"为什么收不到消息"的黑盒里。诚实的约束 > 虚假的简单。

3. **先 200、后台异步跑、主动推回。**
   企业微信要求 5s 内响应回调，而一轮专家可能跑几十秒。被动回复满足不了，
   所以选择"立即 ack + 异步主动发消息"。用户体感是"发完消息，过一会儿专家回你"，
   和真人对话节奏一致，也避免了超时重试导致的重复执行。

4. **多轮上下文复用飞书那套绑定逻辑。**
   不为企业微信单独造会话模型——同一个 `_run_agent_turn` 让两个渠道行为一致，
   未来再加渠道也是同一条路。一致的多轮体验比渠道特性更重要。

## 衡量是否成功
- 用户能在企业微信里发一句、收到专家回答，且第二句能接着上下文。
- 加第三个 webhook 渠道（如公众号）时，几乎只需在后端加一段，不必重写。

## 没做 / 留待以后
- 非文本消息（图片、语音、文件）——先只保文本闭环。
- 流式更新（企业微信被动/主动消息都不便于像飞书卡片那样逐字 PATCH），暂用整段回复。
- 群聊 / 多人场景——目前按单用户 UserID 绑定会话。
