# 为什么需要"原始文件直传"模式

## 用户反馈的痛点
当前所有上传文件都会被自动 OCR / 解析成 markdown 再喂给模型。三类场景里这套默认行为是"善意但错误"的：

1. **盖章 / 签字 PDF 审核**：用户需要的是把整份 PDF 给印章识别 MCP 工具，文本提取出来的"公章（章）"字样毫无意义。
2. **图像 / 视频专用模型**：要把原图喂给视觉模型工具，先 OCR 一次纯属浪费 token，还可能把关键视觉信息抹平。
3. **二进制专用格式**：dwg、psd、parquet 这类文件，mineru 解析后输出的是垃圾文本，反而干扰模型判断。

用户的原话："我上传后这个文件可能直接要给通过智能体发送给 skill 或者 mcp，直接处理原始文件，而不是解析后的内容。"

## 为什么是 Agent 级开关而不是单文件开关
对比三种粒度：

| 方案 | 优点 | 缺点 |
|---|---|---|
| 仅 Agent 级 ✅ | 一次配置长期生效；新建 Agent 时就把语义定下来（"这个智能体专门处理盖章 PDF"） | 同一 Agent 内不能混用 |
| Agent + 单次覆盖 | 最灵活 | UI 复杂、多一处易错点 |
| 仅文件级 | 灵活 | 每次上传都要选一次，重复且容易点错；解析模式不应该是终端用户的认知负担 |

最终选 Agent 级：**解析与否是智能体的能力定义，不是某个文件的属性**。如果以后真的有混合场景，再升级到 "Agent 默认 + 单次覆盖"。

## 为什么 MCP 用签名 URL 而不是本地路径
| 方案 | 适用条件 | 不选的原因 |
|---|---|---|
| 本地路径 | MCP 与 backend 共享文件系统 | 跨主机部署直接破产；MCP 是开放生态，不能假设别人的服务器能访问我的 `/uploads` |
| Base64 内联 | 协议层最干净 | 大文件 prompt 爆炸；多文件场景更糟 |
| **签名 URL** ✅ | 任何 MCP server，只要能发 HTTP | 需要 backend 有公网/内网可达地址；token 短期签发，泄露风险可控 |

签名 URL 是 MCP 生态里事实上的"二进制传递"通用约定（参考 Anthropic 自家 Files API、Workers AI 等都用同款思路）。本地路径作为对 skill 的优化保留——skill 是本地 python，open 文件比走 HTTP 高效得多。

## 为什么后端不自动注入到工具 kwargs
讨论时考虑过：后端拦截工具调用、自动塞 `_attachments=[{...}]` 进 kwargs。最终没选这条路，因为：
- 需要 skill / MCP 工具都按约定字段实现，破坏现有生态
- 模型已经能在 prompt 看到 path/url，让它自己挑哪个工具用、传哪些文件，是更自然的"工具调用"语义
- 隐式注入会让调试链路变黑盒：调用记录里出现一个 user 没写、模型没选的参数

让模型从 prompt 里挑、显式写进 args，既贴近 tool use 的本意，也方便审计。

## 已知局限
- **60 分钟 token 过期**：若模型决定 1 小时后再拉，会 401。当前 chat 流是一次性 SSE，足够；未来若做"长时间后台任务"需要延长或换签发机制。
- **MCP server 必须能访问 backend**：跨网络部署时需要 `BACKEND_BASE_URL` 配公网/同 VPC 地址，否则 URL 不可达。
- **不支持 Agent 内"部分文件解析、部分不解析"**：见上文权衡。

## 未来方向
- 文件级覆盖（Agent 默认 + chat 单次选择），等到用户场景真的需要时再加
- MCP 工具描述里如果声明"需要文件"，前端 chip 可以高亮提示；当前不做，避免引入约定
- 文件级权限审计：哪个 MCP 在什么时候拉了文件，记到 AuditLog
